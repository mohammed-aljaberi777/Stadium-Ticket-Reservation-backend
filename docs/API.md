# API Contract

This document defines the HTTP API surface for the Stadium Ticket Reservation System.

The interactive OpenAPI documentation will be auto-generated by FastAPI at `/docs` once the implementation begins; this file is the canonical, human-readable description.

---

## Design conventions

- **REST-style** resource-oriented URLs.
- **Versioned** with `/v1` prefix on every endpoint.
- **JSON in / JSON out**, `Content-Type: application/json`.
- **Authentication** via JWT `Authorization: Bearer <access_token>` header.
- **Pagination** envelope on list endpoints: `{ items, total, limit, offset }`.
- **Error** envelope on every error: `{ error: { code, message, details } }`.
- **Status codes** used semantically — 200/201/204/400/401/403/404/409/422/429/500.

### Authentication model

- **Access tokens** are short-lived (~15 minutes) and sent on every authenticated request.
- **Refresh tokens** are longer-lived (~7 days) and exchanged at `/v1/auth/refresh` for new access tokens.
- Passwords are stored as bcrypt hashes (cost factor 12).

### Standard error envelope

```json
{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "Too many booking attempts. Try again in 30 seconds.",
    "details": { "retry_after_seconds": 30 }
  }
}
```

### Standard list envelope

```json
{
  "items": [...],
  "total": 137,
  "limit": 20,
  "offset": 0
}
```

### Status codes

| Code | Meaning |
|---|---|
| 200 | OK |
| 201 | Created |
| 204 | No Content (delete or release success) |
| 400 | Bad Request (malformed JSON) |
| 401 | Unauthorized (missing/invalid token) |
| 403 | Forbidden (logged in but not allowed) |
| 404 | Not Found |
| 409 | Conflict (already booked, hold collision, ticket already used) |
| 422 | Unprocessable Entity (validation failure) |
| 429 | Too Many Requests (rate limit) |
| 500 | Internal Server Error |

## Endpoint groups

The API is grouped by user role to make the authorization model obvious:

- **Public** — no authentication
- **Auth** — anonymous, but produces or consumes tokens
- **Fan** — requires `role = FAN`
- **Gate Scanner** — requires `role = GATE_SCANNER`
- **Admin** — requires `role = ADMIN`
- **Health** — for orchestration / readiness probes

## Public endpoints

### `GET /v1/matches`

List upcoming matches.

**Query params:** `status`, `competition`, `from_date`, `to_date`, `limit`, `offset`.

**Layers touched:** Redis (cached), Postgres (on miss).

### `GET /v1/matches/{id}`

Full detail for one match.

### `GET /v1/matches/{id}/sections`

Section summary for a match, used to render the seating-chart overview. Returns `available_seats`, `total_seats`, `min_price`, `max_price` per section.

**Layers touched:** Redis (heavily cached), Postgres (on miss).

### `GET /v1/matches/{id}/seats?section_id=...`

Detailed seat list inside a chosen section. Seats currently held in Redis are surfaced with status `ON_HOLD` so the client never offers an already-locked seat.

**Layers touched:** Postgres + Redis hold check.

### `GET /v1/stadiums` / `GET /v1/teams`

Read-only metadata.

## Auth endpoints

### `POST /v1/auth/register`

Create a new fan account.

**Rate-limited:** 3/min per IP.

**Request:**
```json
{ "email": "anna.mueller@example.com", "password": "...", "full_name": "Anna Müller" }
```

**Response (201):** user object + access_token + refresh_token.

### `POST /v1/auth/login`

Exchange email + password for tokens.

**Rate-limited:** 10/min per IP, 5/min per email.

**Failure modes:** `401` for bad credentials (same message regardless of which field is wrong, to prevent email enumeration). `429` for rate limit. `403` for inactive accounts.

### `POST /v1/auth/refresh`

Exchange refresh token for a new access token.

### `GET /v1/auth/me`

Return the currently authenticated user's profile.

## Fan endpoints

### Holds

#### `POST /v1/matches/{match_id}/holds`

Create a Redis-backed temporary hold on one or more seats.

**Anti-scalping checkpoints:**
- Sliding-window rate limit (5/min user, 10/min IP)
- Hold cap: active holds + confirmed tickets must not exceed `max_tickets_per_user`

**Layers touched:** Redis (atomic `SET … NX EX 300` per seat) + Postgres (read for availability check).

**Request:**
```json
{ "match_seat_ids": ["ms-uuid-1", "ms-uuid-2"] }
```

**Response (201):**
```json
{
  "hold_id": "hold-uuid",
  "match_id": "match-uuid",
  "match_seat_ids": ["ms-uuid-1", "ms-uuid-2"],
  "expires_at": "2026-09-12T10:05:00Z",
  "seconds_remaining": 300
}
```

**All-or-nothing semantics:** if any seat is already held or booked, the whole request rolls back and returns `409`.

#### `DELETE /v1/holds/{hold_id}`

Release a hold early.

**Response (204):** No content.

#### `GET /v1/holds/me`

List the current user's active holds with seat metadata and time remaining.

### Bookings

#### `POST /v1/bookings`

**The** booking endpoint. Converts a Redis hold into a permanent booking with tickets and QR codes.

**Anti-scalping checkpoints:** all four (rate limit, cooldown, per-match cap re-check, personalized QR).

**Request:**
```json
{ "hold_id": "hold-uuid", "payment_method_token": "pm_tok_simulated" }
```

**Response (201):** booking object + tickets array (each with `qr_url`).

**See the [17-step booking flow](#the-17-step-booking-flow) below for the full layered defense.**

#### `GET /v1/bookings/me`

Paginated list of the current user's bookings.

#### `GET /v1/bookings/{id}`

Single booking detail (owner or admin only).

#### `POST /v1/bookings/{id}/cancel`

Self-cancellation. Allowed only when:
- Booking status is `CONFIRMED`
- Match is more than 48 hours away from kickoff

Runs as a single Postgres transaction: marks booking `CANCELLED`, tickets `REVOKED`, returns `match_seats` to `AVAILABLE`.

### Tickets

#### `GET /v1/tickets/me`

List the current user's tickets.

#### `GET /v1/tickets/{id}`

Single ticket detail.

#### `GET /v1/tickets/{id}/qr`

Return the QR code PNG. The QR encodes a signed JWT containing `ticket_id`, `match_id`, `user_id`, and a timestamp.

## Gate Scanner endpoint

### `POST /v1/verify`

Scan and consume a ticket at the stadium gate.

**Auth:** requires `role = GATE_SCANNER`.

**Request:**
```json
{ "qr_token": "eyJhbGciOiJIUzI1Ni..." }
```

**Response (200, approved):**
```json
{
  "result": "APPROVED",
  "ticket": {
    "id": "...",
    "user_full_name": "Anna Müller",
    "section": "Nordkurve",
    "row": "12",
    "seat": "7",
    "match": "FC Bayern München vs Borussia Dortmund"
  }
}
```

**Response (409, rejected):**
```json
{
  "result": "REJECTED",
  "reason": "ALREADY_USED",
  "details": { "used_at": "2026-09-12T17:42:11Z" }
}
```

**The atomic core:**

```sql
UPDATE tickets
SET status = 'USED',
    used_at = NOW(),
    scanned_by_user_id = $scanner_id
WHERE id = $ticket_id AND status = 'ISSUED'
RETURNING *;
```

The `WHERE status = 'ISSUED'` clause makes the entire "check + consume" operation a single atomic database step. Two scanners scanning the same QR simultaneously? Only one update succeeds; the other returns an empty `RETURNING` set and rejects.

## Admin endpoints

All admin endpoints require `role = ADMIN`. Failure: `403`.

| Endpoint | Purpose |
|---|---|
| `POST /v1/admin/stadiums` | Create a stadium |
| `POST /v1/admin/stadiums/{id}/sections` | Add a section |
| `POST /v1/admin/sections/{id}/seats` | Bulk-create seats |
| `POST /v1/admin/teams` | Create a team |
| `GET /v1/admin/teams` | List teams |
| `POST /v1/admin/matches` | Create a match |
| `PATCH /v1/admin/matches/{id}` | Update match (status, sale window, ticket cap) |
| `POST /v1/admin/matches/{id}/inventory` | Generate `match_seats` rows from section pricing |
| `GET /v1/admin/matches/{id}/stats` | Sold counts, revenue |
| `POST /v1/admin/bookings/{id}/refund` | Refund a single booking |
| `POST /v1/admin/matches/{id}/refund-all` | Cancel match + refund all bookings (one transaction) |
| `GET /v1/admin/users` | Paginated user list with filters |
| `PATCH /v1/admin/users/{id}` | Change role or deactivate |

### Inventory generation request shape

```json
{
  "section_prices": [
    { "section_id": "...", "price": "75.00" },
    { "section_id": "...", "price": "120.00" }
  ]
}
```

Generates one `match_seats` row per seat in each priced section, in a single SQL `INSERT … SELECT` per tier.

### Refund transaction

```sql
BEGIN;
UPDATE bookings    SET status = 'REFUNDED' WHERE id = $1;
UPDATE tickets     SET status = 'REVOKED'  WHERE booking_id = $1;
UPDATE match_seats SET status = 'AVAILABLE'
  WHERE id IN (SELECT match_seat_id FROM tickets WHERE booking_id = $1);
COMMIT;
```

## Health endpoints

### `GET /health`

Liveness probe. Returns `200 { "status": "ok" }` if the process is alive. **Does not touch Postgres or Redis** — by design.

### `GET /health/ready`

Readiness probe. Verifies Postgres (`SELECT 1`) and Redis (`PING`).

```json
{
  "status": "ready",
  "checks": { "postgres": "ok", "redis": "ok" }
}
```

Returns `503` with the failing dependency noted if either is down.

## The 17-step booking flow

This is the most important sequence in the API. The ordering matters; each step protects a specific risk.

| # | Step | Layer | What it protects against |
|---|---|---|---|
| 1 | JWT auth check | Middleware | Anonymous abuse |
| 2 | Rate-limit check (5/min user, 10/min IP) | Redis | Bot scripts |
| 3 | Cooldown check (30s post-success) | Redis | Rapid-fire scripts |
| 4 | Verify hold exists, owned by user, not expired | Redis | Hijacking another user's hold |
| 5 | Resolve `hold_id` → list of `match_seat_id`s | Redis | — |
| 6 | `BEGIN` transaction | Postgres | — |
| 7 | `SELECT … FOR UPDATE` on those `match_seats` | Postgres | Concurrent booking of same seat |
| 8 | Verify each row is still `AVAILABLE` | Postgres | Stale hold edge case |
| 9 | Count user's existing CONFIRMED tickets for this match | Postgres | Per-match scalping cap |
| 10 | If (existing + new > `max_tickets_per_user`) → `ROLLBACK` | Postgres | Multi-account / Sybil bypass |
| 11 | Process payment (mocked in v1) | External | — |
| 12 | `INSERT` into `bookings`, `tickets` | Postgres | — |
| 13 | `UPDATE match_seats SET status = 'BOOKED'` | Postgres | — |
| 14 | `COMMIT` | Postgres | All-or-nothing |
| 15 | `DEL` the Redis hold keys | Redis | Free hold tracking |
| 16 | `SET cooldown:booking:user:<id>` 30s | Redis | Throttle next attempt |
| 17 | Return booking + tickets to client | — | — |

The critical insight: step 9 counts existing tickets **inside the transaction, after the seat rows are locked**. A bot firing 10 simultaneous requests can hold the seat-row lock only one at a time. The second request sees the updated count and `ROLLBACK`s. The race is mathematically closed.

---

## Anti-scalping checkpoint summary

| Defense | Where it triggers | What it protects |
|---|---|---|
| Rate limit (5/min user, 10/min IP) | Middleware on `POST /holds`, `POST /bookings`, `POST /auth/login` | Bot scripts |
| 30s cooldown after success | Redis SET after successful `POST /bookings` | Rapid-fire scripts |
| `max_tickets_per_user` cap | Redis at hold time + Postgres inside booking transaction | Bulk single-account purchases |
| Personalized QR (signed JWT with `user_id`) | Embedded at `POST /bookings`; surfaced at `POST /verify` | Resale to strangers, gate-side fraud |

Defense-in-depth: cheap filters (rate limit, cooldown) run first; the expensive transactional check is the last line of defense.

## End-to-end fan journey

```
Browse:        GET    /v1/matches?status=ON_SALE
               GET    /v1/matches/{id}/sections
               GET    /v1/matches/{id}/seats?section_id=...

Select:        POST   /v1/matches/{id}/holds         ← Redis hold
               (5 minutes for payment)

Confirm:       POST   /v1/bookings                   ← layered defense + Postgres TX

View ticket:   GET    /v1/tickets/me
               GET    /v1/tickets/{id}/qr            ← PNG with signed JWT

Matchday:      POST   /v1/verify                     ← atomic UPDATE; entry approved
```
